### ğŸ”¥ Example of Clean Pseudocode (Binary Search)

```
Function binarySearch(arr, target)
    Set low = 0
    Set high = length of arr - 1

    While low <= high
        Set mid = (low + high) / 2

        If arr[mid] == target
            Return mid

        Else if arr[mid] < target
            Set low = mid + 1

        Else
            Set high = mid - 1

    Return -1  // Target not found

```

## ğŸ“ Ultimate Algorithm Development Checklist

## 1. **Understand the Problem (NO coding yet)**

- ğŸ”² Read the problem _twice_ slowly.
    
- ğŸ”² Identify **Inputs** and **Outputs**.
    
- ğŸ”² Clarify **Constraints** (size limits, value limits, etc.).
    
- ğŸ”² Note **Edge Cases** (empty inputs? negatives? duplicates?).
    

---

## 2. **Plan a Solution (on Paper / Mind)**

- ğŸ”² Write a **brief English explanation** of what you think the solution idea is.
    
- ğŸ”² Draw a **small example manually** (even 3â€“5 elements).
    
- ğŸ”² If complex, **draw a flowchart** or **state diagram**.
    
- ğŸ”² Think: Is it **brute force**? Can it be **optimized**?
    

---

## 3. **Write Pseudocode (clean and readable)**

- ğŸ”² Use plain English mixed with structured keywords (`if`, `while`, `for`).
    
- ğŸ”² Focus on **logic**, not syntax.
    
- ğŸ”² Indent steps properly.
    
- ğŸ”² Add 1â€“2 short **comments** explaining key parts.
    

---

## 4. **Analyze Complexity (Big-O Thinking)**

- ğŸ”² What's the **Time Complexity**? (O(n), O(nÂ²), O(log n)...)
    
- ğŸ”² What's the **Space Complexity**?
    
- ğŸ”² Is it acceptable for given problem constraints?
    

---

## 5. **Code the Solution (only now!)**

- ğŸ”² Translate pseudocode into real code **carefully**.
    
- ğŸ”² Use small, meaningful variable names.
    
- ğŸ”² Keep functions **small** and **focused**.
    

---

## 6. **Test Thoroughly**

- ğŸ”² Test with **small examples** first.
    
- ğŸ”² Test with **large inputs** if possible.
    
- ğŸ”² Test **edge cases** (empty, huge numbers, wrong formats).
    

---

## 7. **Reflect and Optimize**

- ğŸ”² Could the solution be **simpler** or **faster**?
    
- ğŸ”² Is memory usage **reasonable**?
    
- ğŸ”² Could recursion become iteration (if stack overflow risk)?
